Http Requests Monitoring

Monitor your application behaviour during incoming http requests.

Introduction

¶

To activate inspection when your application is executed by an incoming http request you can use the WebRequestMonitoring middleware.

Thanks to the middleware you are free to decide on which routes you want activate monitoring, based on your routes configuration or on your personal monitoring preferences.

WebRequestMonitoring middleware works like any other Laravel middleware you are familiar to.

To get more information on how middlewares works in Laravel, take a look at Laravel's official documentation.

Option 1 - Attach to the middleware group (recommended)

¶

Basically you can attach the middleware in the App\Http\Kernel class in one or more of your predefined middleware groups:

1/**
2 * The application's route middleware groups.
3 *
4 * @var  array
5 */
6protected $middlewareGroups = [
7    'web' => [
8        ...,
9        \Ultimate\Laravel\Middleware\WebRequestMonitoring::class,
10    ],
11
12    'api' => [
13        ...,
14        \Ultimate\Laravel\Middleware\WebRequestMonitoring::class,
15    ]
16]

Usually web and api middleare groups wrap your entire application, so in two lines of code you can monitoring all incoming http requests.

Option 2 - Use as middleware key

¶

In alternative you can create a middleware key so you can attach it in a specific route or group:

1/*
2 * Create a new middleware key (ultimate) in your \App\Http\Kernel class.
3 */
4protected $routeMiddleware = [
5    ...,
6    
7    'ultimate' => \Ultimate\Laravel\Middleware\WebRequestMonitoring::class,
8];

Use "ultimate" middleware key in you routes:

1/*
2 * Attach the "ultimate" middleware in your routes
3 */
4Route::prefix('app')->middleware('ultimate')->group(function () {
5    ...
6});

Ignore Web Requests

¶

Filtering by URLs is supported by default via a simple configuration parameter. Learn more here.

Palzin APM provide a basic strategy to turn off monitoring in certain part of your application, but the middleware is designed to allows you to extend it and add your own strategy overwriting shouldRecorded method that should return a boolean to determine if the current request should be recorded or not.

Run the artisan command below to create a new middleware class:

1php artisan make:middleware UltimateFilterMonitoringMiddleware

In the new middleware class extend the Palzin APM middleware and override the shouldRecorded method to implement your filtering strategy:

1<?php
2
3use \Ultimate\Laravel\Middleware\WebRequestMonitoring;
4
5
6class UltimateFilterMonitoringMiddleware extends WebRequestMonitoring
7{
8    /**
9     * Determine if Palzin APM should record current request.
10     *
11     * @param \Illuminate\Http\Request $request
12     * @return bool
13     */
14    protected function shouldRecorded($request): bool
15    {
16        return $request->userAgent() === 'Reliable User Agent';
17    }
18}

If the method return true the web transaction will be recorded in your dashboard, if it return false the transaction will be completely ignored.

Remember to attach your new UltimateFilterMonitoringMiddleware class to the App\Http\Kernel:class instead of the default Palzin APM middleware.

Hide sensible contents

¶

You may hide sensible data from HTTP reuqests body like passwords. Palzin APM is able to detect that parameters in your request's body masking their content with a simple "**".

Simply add fields to the hidden_parameters array in the in Palzin APM config file:

1'hidden_parameters' => [
2    'password',
3    'password_confirmation',
4    
5    // Other fields here...
6],

You can specify nested fields using the dot notation like user.password

Identify Controller@Method execution

¶

If you prefer to have a clear visibility of what controller and method are executed during an HTTP request instead of the request path, you can use the code below inside the boot method of one of your ServiceProvider.

Maybe the best place could be EventsServiceProvider.

1use Illuminate\Routing\Events\RouteMatched;
2
3/**
4 * Register any events for your application.
5 *
6 * @return void
7 */
8public function boot() 
9{
10    parent::boot();
11
12    /*
13     * Changes the way Palzin APM reports transactions in your dashboard.
14     */
15    Event::listen(RouteMatched::class, function(RouteMatched $event) {
16        $name = $event->route->getActionName();
17    
18        if(ultimate()->hasTransaction()) {
19            ultimate()->currentTransaction()->name = $name;
20        } else {
21            ultimate()->startTransaction($name);
22        }
23    });
24}

This will change the way Palzin APM reports transactions in your dashboard from GET /path/{id} to YourController@yourMethod .
